# 序

https://github.com/marudedameo2019/atomic_fallback_examples/blob/main/data_race.cpp

例えばこんなデータ競合が起こるケース。10個のスレッドから100000回ずつインクリメントされて、期待値1000000な`count`だけど、実際には随分と少なくなります。

```shell-session
$ g++ -g -O3 -Wall -pedantic data_race.cpp -o data_race
$ ./data_race
expected count: 1000000
actual count:   356107
$ 
```

これはざっくり言うと、`count++`の処理、「値を読んで、1つ足して、また書き込む」、が複数のスレッドから同時に行われたために、同じ値に1足して同じ値を書き込んでるときがあったりするからです。ようは`count++`の処理は複数のスレッドから完全に同時には行えないということです。

ちゃんと動かすにはこうしないといけません。差分で言うとこんな感じ…

```diff cpp
@@ -1,12 +1,13 @@
 #include <iostream>
 #include <thread>
 #include <vector>
+#include <atomic>

     constexpr int INCREMENT_COUNT = 100000;
     constexpr int THREAD_COUNT = 10;

-    volatile int count = 0;
+    std::atomic<int> count = 0;
     auto increment = [&]{
         for (int i = 0; i < INCREMENT_COUNT; ++i) count++;
```

全文掲載するとこんな感じです。

https://github.com/marudedameo2019/atomic_fallback_examples/blob/main/data_atomic.cpp

こうすると、countは原則ちゃんとインクリメントされ、期待値どおりの値が出力されるようになります。
これが[std::atomic](https://cpprefjp.github.io/reference/atomic/atomic.html)なんですが、実際には`count++`を(x86-64なら^[説明は多分x86が一番楽なので以降もx86を前提とします])

```asm
    lock add DWORD PTR [rdx],0x1
```

のようなコードにしてくれる機能になります。ようはstd::atomicとは、OSが用意するmutexのような重たい同期機能を使用せず、原則CPU命令を使った原始的な同期機能を使用したいときに使うものなのです。

ただ`std::atomic<T>`のように汎用に書いてしまうと、`T`が`int`のような小さい(4バイト)値ならCPU命令で何とかなりそうですが、大きくなったらどうなるのか気になります。今回は`std::atomic<T>`の`T`が、もっと大きな値(16byteなど)だったときどうなるのか？というお話です。

# 1. 前提と対象読者

- 環境はx86-64なPC上のLinuxです(Ubuntu 24.04で見ていきます)
- C++を知っている方
- binutilsやgdbの使用に抵抗がない方
- x86-64アセンブラもある程度読める方
- mutexなどの同期機構を使ったことがある方

`std::atomic<__int128>`に`++`演算子なんてないよ、とコンパイルエラーが出ています。元々定義されている整数型では全て`++`演算子が用意されているのですが、`__int128`はコンパイラの拡張型であり、本来のC++には存在しない型だからです^[`__int128`自体には`++`が使えますが、`std::atomic<__int128>`には`++`が使えない、という話なので注意]。なお、x86-64においてはそれだけでなく、128bitでは1命令でlockしながらインクリメントすることができません。なので、`++`演算子の代わりに`fetch_add()`を使っても結果は同じになります。

## 2-3. まとめ

- `++`インクリメント演算子を使用する実装では、8バイトまでしかコンパイルできない

# 3. CASループでインクリメント

`++`が使えないので先人の知恵、CAS([Compare and Swap](https://en.wikipedia.org/wiki/Compare-and-swap))ループでインクリメントしてみます。

## 3-1. 実装

https://github.com/marudedameo2019/atomic_fallback_examples/blob/main/compare_exchange.h

https://github.com/marudedameo2019/atomic_fallback_examples/blob/main/integer_compare_exchange.cpp

一度loadして、比較してまだ同じなら+1した値とswapして終了、違ってたらロードした値を更新して再挑戦、というコードになっています。つまりatomicにincrementするのではなく、atomicな操作を何個かに分けて成功するまで挑戦するというコードです。肝は[compare_exchange_weak](https://cpprefjp.github.io/reference/atomic/atomic/compare_exchange_weak.html)になります^[正直CASループを入れるくらいなら、mutexの方が個人的には好みです]。

`__atomic_load_16`や`__atomic_compare_exchange_16`が見つからないと言われています。すぐ分かるとおり、これらは`std::atomic<__int128>`で必要になっている関数です。それぞれ、`load()`や`compare_exchange_weak()`で必要になっています。ただし`std::atomic<int32_t>`や`std::atomic<int64_t>`はそういう関数を必要としていません。x86-64ではそれらの処理がCPUの1命令で書けるからです。そして、上のリンクエラーで必要とされる関数は、エラー回避(fallback)の目的で使用される、CPUの違いを吸収した汎用の関数です^[もちろん最悪時にはmutexを使ったりもするので、遅くなります]。

それらの関数は[libatomic](https://github.com/gcc-mirror/gcc/tree/master/libatomic)に用意されているので`-latomic`を加えます。

```sh
$ g++ -g -std=c++17 -O3 -Wall integer_compare_exchange.cpp -o integer_compare_exchange -latomic
$ ./integer_compare_exchange
sizeof(T): 4
expected count: 1000000
actual count:   1000000
sizeof(T): 8
expected count: 1000000
actual count:   1000000
sizeof(T): 16
expected count: 1000000
actual count:   1000000
$ 
```

ようやく16バイトのカウントでも正しく同期を取りながらカウントアップできるようになったようです。
しかし、これらは実際どんなコードに落ちているのでしょう？気になります。

## 3-4. まとめ

- CASループを使うと16byte整数でもstd::atomicを使用してインクリメントできる
- 16byte整数ではlibatomicが用意したfallback関数を使用する
- fallback関数は実行時のCPUIDから適応的に選択される
- fallback関数は実行時のCPUIDから適応的に選択されるが、その処理の中には**mutexを使うものがある**
- fallback関数はインライン展開できないので、1命令しか処理がなくても遅そう

`__atomic_load`の第一引数はサイズなので、`0x4075bf`の判定で0x10(16)より大きいためすぐに`0x407675`に飛びます。するとすぐに`libat_lock_n`が呼ばれることになります。見れば分かるとおり、この関数は内部で`___pthread_mutex_lock`を呼んでいます。つまり、std::atomic<s32>::load()時にmutexロックされるということです。

## 5-4. まとめ

- 構造体を使った場合4/8バイトのデータなら整数CASループと遜色ない速度になる
- 構造体を使った場合16バイトのデータだとfallback関数は整数CASループと同じものが使用される
- 構造体を使った場合16バイトのデータだと整数CASループより何割か速度が落ちる
- **構造体を使った場合32バイトのデータだとmutexが使用される**
- 構造体を使った場合32バイトのデータだと整数CASループの16バイトの2～3倍程度の時間がかかる

# 6. 全体まとめ

- std::atomicは`++`演算子を8バイト整数までしか使えない
- 今回の例でstd::atomicの`++`演算子はCASループを使ったインクリメントより4倍速かった
- **std::atomicは中身がデカいとfallback関数を使用してmutexロックすることがある**
- fallback関数はCPUIDを使用して適応的に実装を選択している
